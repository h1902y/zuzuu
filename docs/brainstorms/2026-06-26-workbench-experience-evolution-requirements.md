# Workbench Experience Evolution — make the loop legible, the experience attention-first

## Summary

A phased program over the zuzuu workbench (`web/src/client/`) that makes the **observe → propose → review → evolve** loop legible and the experience attention-first. The review gate is the product's moat but is under-exercised: its payoff moment is silent, blind, and a chore. Seven directions (from the ce-ideate run) sequence into three phases — **P1 legibility**, **P2 the bold reframes** (behind toggles / opt-in), **P3 experience & delight**. Every change reduces gate *friction*; none removes the gate.

## Problem Frame

zuzuu's value is the per-project brain it grows from real coding sessions, gated by human review. The workbench chrome is polished, but the loop's payoff is invisible: a proposal arrives with no stated reason and no preview of what lands, so the rational response is reject-everything (or rubber-stamp) — and the loop dies after the first session. Observation runs silently at session-end, so the user never feels watched or sees "look what I taught it." The launcher scales to ~3 projects, then becomes a graveyard of timestamps. Five independent ideation frames converged on one fault line: the gate's burden, the trust it demands, and the Work ⇄ Brain perception gap. This program pushes on that spine.

## Key Decisions

- **The gate stays a gate everywhere.** This program reduces friction *around* review; it never removes the human gate. Auto-promotion (P2) is scoped so the gate's protective role is untouched.
- **Reuse existing seams; no new backend service.** The data these features need already exists: the `ROUTE` table in `grow/observe.mjs` computes *why* each proposal exists; generations are content-addressed (cheap before/after diffs and one-click undo); the append-only log is chronological (the feed + provenance); session lifecycle hooks already fire. The work is surfacing discarded data, not generating new data.
- **Bold reframes ship behind toggles, not as forced replacements.** The attention-feed (R7) and the brain-feed (R16) are new default lenses with the familiar grid/launcher one toggle away — never removed.
- **Auto-promotion is conservative by construction.** Default everything to must-decide; auto-promote is opt-in per module, never applies to destructive/irreversible/protective writes (guardrails are always must-decide), and is always one-click reversible.
- **Phasing is the build order.** P1 ships the convergent, low-risk legibility wins first; P2 and P3 are gated on P1 landing. Each phase is independently shippable.

## Requirements

R-IDs are continuous across phases. Each phase is independently shippable; later phases assume earlier ones.

### P1 — Legibility (ship first; mostly reads of existing data)

**The legible gate (idea #1 — the spine)**

- **R1.** Every proposal card shows a plain-English **reason line** derived from the `ROUTE` rule that produced it (e.g., "Because you ran `rm -rf dist` in 3 sessions, I want to add a confirm-guardrail").
- **R2.** Every proposal card shows a **before/after diff** of the note body, read from the content-addressed generation store; long diffs collapse by default.
- **R3.** Every proposal card carries a **type chip** (Knowledge / Memory / Actions / Instructions / Guardrails) so a protective ask reads visibly differently from a low-stakes fact.
- **R4.** The reason line is generated by a `ROUTE`-kind → prose templating layer, one template per signal kind; copy is treated as load-bearing and reviewed.

**Session-close bridge + provenance (idea #4)**

- **R5.** When a session transitions live → ended with pending proposals > 0, surface a **"what this session taught"** card (count by type, the top mined patterns, a "Review now" action) at the close moment; the card is dismissible.
- **R6.** Every note exposes a **"born here"** link to the exact transcript moment (command / correction / file-edit) `observe` mined it from, plus a bidirectional session ↔ proposals cross-reference (a session shows the proposals it bore; a note shows its originating session).

### P2 — The bold reframes (behind a toggle / opt-in; gated on P1)

**Attention-feed landing (idea #2)**

- **R7.** Add an **Attention** view to the Projects Home (toggle: Projects ⇄ Attention): a bounded, sorted, cross-project stream of pending proposals, just-ended sessions, and stale-brain flags, with a **"caught up"** terminal state. The project grid remains as the alternate view, never removed.
- **R8.** The Attention view leads with a **"while you were away"** summary on return (sessions observed, proposals grown, modules changed since last visit) and per-item resume context (what the pending items are, which session was live).

**Confidence-tiered review (idea #3 — the riskiest; ship conservative)**

- **R9.** Sort proposals into three lanes by confidence × reversibility: **auto-promoted** (an "already promoted — undo?" lane), **must-decide** (the human gate), and **hold** (borderline; re-surfaces only if corroboration climbs).
- **R10.** Auto-promotion is **opt-in per module, default off**, and **never** applies to destructive, irreversible, or protective writes — guardrail proposals are always must-decide.
- **R11.** Every auto-promoted item is **reversible in one click** from its lane (a gated rollback proposal, not a silent mutation).
- **R12.** A rejected proposal leaves a **rejection memory**, so `observe` re-mining the same signal does not re-propose it identically; the user can escalate a rejection to "don't propose this again."

### P3 — Experience & delight (gated on P1/P2)

**Setup as the brain's first proposals (idea #5)**

- **R13.** Replace the linear setup Stepper with the project's **day-zero proposals reviewed through the same gate** ("protect against secret reads · watch `app.tsx` · enable observe"); each remains a **visible consent**, never silent.
- **R14.** The proposal that enables host hooks explains, in its copy, what enabling does (it is a real change to the host).
- **R15.** After the first real session, hand off to a **narrated "here are the first 3 things I noticed about your code"** — real mined proposals, not synthetic demo copy.

**Brain as a "what I learned" feed (idea #6)**

- **R16.** Add a **feed lens** to the Brain world as a new default: a reverse-chronological stream of learning-cards ("Learned: `app.tsx` is the hot file · 3 sessions ago"), **corroboration-weighted** so load-bearing notes read bolder. The grid/table view stays one fast toggle away as the power lens.

**Identity from the brain (idea #7)**

- **R17.** Derive and **propose** a project's emoji / color / one-line descriptor from what its brain has learned, ratified through the gate; the user can override trivially. Identity becomes an evolving reviewed artifact, not a creation-time input.

## Scope Boundaries

**Deferred within this program:** P2 and P3 are gated on P1 shipping and proving the legibility wins. R9–R11 (auto-promotion) ship only after the must-decide + diff + undo surfaces (P1) are solid.

**Outside this product's identity (not this brainstorm):** a **team-shared brain** with attributed proposals and delegated review. The git-native substrate (sessions = branches, brain = repo) makes it nearly free, but it expands the single-user workbench into a team product — its own ideation.

**Unchanged:** the loop, the gate-as-moat, git-native operation (no live service), sessions = git branches, the daemon-shells-CLI model, design-system-first composition (`web/src/client/ds/`, the `ds-no-inline` guard).

## Success Criteria

- **The loop survives past session one.** A new user reaches and acts on their first real proposal in the setup session (R13–R15), and keeps reviewing across subsequent sessions rather than abandoning (R1–R6).
- **Review is a real decision, not a reflex.** Proposal cards let a user judge from the reason + diff + type without opening anything (R1–R3); approval accuracy improves in both directions.
- **The gate scales past 3 projects.** A multi-project user triages from the Attention view and reaches a "caught up" state (R7–R8) instead of scanning a grid of timestamps.
- **The Work ⇄ Brain gap closes at the seam.** A finished session produces a felt "look what I taught it" moment (R5), and any note traces to its origin in one jump (R6).
- **The moat holds.** No destructive/protective write is ever auto-applied (R10); every auto-promotion is reversible (R11).

## Dependencies, Assumptions & Outstanding Questions

- **Session-close event (R5).** Needs a daemon → client signal on the live → ended transition. Assumption to verify in planning: the lifecycle hooks fire today but are discarded client-side — the plumbing is new.
- **Provenance source-span (R6).** Needs `observe` to persist the transcript offset a note was mined from. Open: does it record this today, or is persistence new work?
- **Confidence score (R9).** Needs a concrete definition — corroboration count × a reversibility class per module kind. Deferred to ce-plan; this brainstorm fixes the *behavior* (three lanes, conservative defaults), not the scoring formula.
- **Reason templating (R4).** Assumes every `ROUTE` kind can produce a faithful one-line rationale; kinds without a clean template fall back to a neutral "recurring signal" line rather than a misleading one.
- **Toggle persistence (R7, R16).** The Projects ⇄ Attention and grid ⇄ feed lens choices should persist per user; where (client store vs config) is a planning detail.
