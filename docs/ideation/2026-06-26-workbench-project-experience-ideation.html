The Workbench Project Experience

Date 2026-06-26  ·  Topic workbench-project-experience  ·  Focus the workbench project experience (Projects Home · project page · navigation · setup→working)  ·  Mode repo-grounded

Five ideation frames ran independently over this surface. They converged — hard — on one place: the review gate. Pain, inversion, leverage, cross-domain analogy, and assumption-breaking all, separately, named the gate's burden, the trust it asks for, and the Work ⇄ Brain perception gap as the experience's real fault line. That convergence is the headline finding: the workbench's chrome is polished; its moat is under-exercised because the loop's payoff moment is silent, blind, and a chore. The seven survivors below all push on that.

Grounding Context

Subject. The browser workbench (web/src/client/) a local daemon serves: landing, project selection, orientation, setup, and working. zuzuu grows a per-project brain (modules of notes) from real coding sessions via observe → propose → review → evolve, every write human-gated (the review gate is the moat). Sessions = git branches. Locked vision: CRUD-to-app over Project-as-DB, two worlds Work (terminal/sessions) ⇄ Brain (notes/modules dashboard).

Built & current. Projects Home (ProjectsHome.tsx: registry-backed list, emoji+health rows, search/sort/group); the Stage+Wings project page (WorkbenchShell.tsx: header picker, NavTree, stage = terminal/grid/record/graph/overview, a review/form/schema wing, the Ribbon footer with "press R to review"); a guided setup Stepper (git-init→init→enable→session→review); the Overview home base (SESSIONS ⇄ THE BRAIN); the review gate; per-module generations + rollback; per-project emoji + theme; browser back/forward tied to home↔project + path.

Load-bearing facts the ideas exploit. grow/observe.mjs holds a declarative ROUTE table that already computes why each proposal exists (recurring command → action; repeated destructive command → ask-guardrail) — and the UI discards it. Generations are content-addressed; rollback = pointer-flip (most writes are cheaply reversible). The brain is git-native and portable. Observe runs at session-end, untended. zz init ships only a safety floor; modules materialize on demand. The "review" rung has no CTA — first value is gated on real observed work (cold-start).

Verification note: load-bearing direct: bases were checked against first-hand knowledge of the cited files (authored/modified this session) rather than a fresh verifier — the lower-bias path here; reasoned:/external: bases arbitrated directly.

1The legible gate — every proposal states its reason and shows its diff

TOP PICKA4 · working-loopConfidence 90%Complexity Medium
DescriptionToday a proposal card asks for a yes/no with almost no information: the user can't see why zuzuu wants this or what exactly lands if they approve. Make every card self-explaining: (a) a plain-English headline drawn from the ROUTE rule that produced it — "Because you ran rm -rf dist in 3 sessions, I want to add a confirm-guardrail"; (b) a before/after generation diff of the note body, pulled from the content-addressed store; (c) a small type chip (Knowledge / Guardrail / Instruction) so a safety-relevant ask reads differently from a low-stakes fact. The gate stops being a blind consent and becomes an actual decision.
Basis
direct grow/observe.mjs's declarative ROUTE table already maps each mined signal → module + a reason string (it produced the proposal); the UI never surfaces it. Generations are content-addressed, so a before/after diff is a read the daemon can already serve. external Recommender "Because you watched X" rationale lines as the trust-through-explanation pattern (Netflix/Spotify).
RationaleThis is the cheapest, highest-leverage lever in the set and it sits squarely on the moat. A proposal with no stated reason is a demand for blind trust, and the rational response is the reject-everything reflex — which kills the loop. The data already exists and is thrown away; surfacing it turns each review from "do I trust this black box?" into "do I agree with this stated reasoning?" — a faster, higher-quality judgment that raises approval accuracy in both directions.
DownsidesThe diff view adds card height (manage with collapse-by-default for long bodies); the ROUTE→prose translation needs a small templating layer per signal kind; a poorly-worded rationale could mislead, so the copy is load-bearing.

2The attention-feed landing — the launcher becomes a "what needs me" queue

A1 · selection & launchConfidence 70%Complexity High
DescriptionInvert the Projects Home from a launcher into a single cross-project attention feed: a unified, bounded stream of every project's pending proposals, just-ended sessions, and stale-brain flags, sorted by what needs a human — with a satisfying "caught up" terminal state. You descend into a full Project page only when an item needs context. The project grid still exists, demoted to a filter/secondary view. This folds in the warm-return digest ("4 sessions observed, 9 proposals while you were away"), per-row resume context, and group-level triage.
Basis
reasoned The product's scarce resource is human gate decisions, not folder navigation — every project accrues proposals in the background via observe. A launcher organized by project forces the user to remember which of N projects owes review (the surface itself admits review is "something users must remember to do, press R"). Organizing the landing by attention owed matches the system's real output, and it's the only model that survives past ~3 projects. external Anki's bounded "due today" + "0 left" rhythm as the proven anti-backlog mechanism.
RationaleChanges what the user does first every day — from "navigate to a project" to "clear what the brains grew." It makes the gate the front door, which is also the moat, and converts the loop's burden from an ever-climbing dread-counter into a daily ritual with an end.
DownsidesA bold reframe of the product's identity surface — the project grid is a familiar, reassuring metaphor; burying it risks disorientation for new users. Needs the per-project pending/session signal on the cross-project list payload (already mostly present). Best shipped behind a "Home view: Projects / Attention" toggle first.

3Confidence-tiered review — the brain reviews itself; you ratify

A4 · working-loopConfidence 75%Complexity High
DescriptionStop treating every proposal as an equal-weight human decision. Score each by confidence × reversibility and route it: clearly-good, cheaply-reversible ones (a filepath fact corroborated 10×) auto-evolve into an "already promoted — undo?" lane; ambiguous or destructive ones (a new guardrail, a table split) sit in a must-decide lane; borderline-but-rising ones get a third verb — hold — and re-surface only if corroboration climbs (instead of forcing a premature coin-flip). Rejection leaves a memory so the same signal isn't re-proposed identically.
Basis
reasoned A corroborated-10× filepath fact and a new destructive guardrail carry wildly different risk, yet the gate weights them identically — that's what makes humans rubber-stamp and stop reading. Generations make most writes cheaply reversible ("rollback = flip pointer"), so "approve" is recoverable for the easy class; the gate's value concentrates on the irreversible/protective writes. external The newsroom wire-desk's third verb (run / hold / spike) — separating arrival from decision so borderline signal isn't lost.
RationaleDirectly attacks "review, never-as-a-chore" without abandoning the gate — it reallocates human attention to the decisions that are actually decisions. The reject-with-reason stream and corroboration counts the loop already produces are exactly the training signal the confidence model needs.
DownsidesThe riskiest idea to get wrong — auto-promote on the wrong class erodes the moat and trust. Must ship conservative (default everything to must-decide; auto-promote is opt-in per module, and the "undo" lane must be one click). A miscalibrated confidence score is worse than none. Pairs hard with idea #1 (the diff is what makes "undo?" safe).

4The session-close bridge + provenance threads

A4 · working-loopConfidence 85%Complexity Medium
DescriptionTwo halves of one move that closes the Work⇄Brain gap. (a) Session-close brief: when a session transitions live→ended with pending > 0, surface a lightweight card — "this session generated 3 proposals: 2 actions, 1 guardrail, from debugging the auth flow" + a "Review now" CTA — at the exact reflective moment the user just finished work. (b) Provenance threads: every note carries a "born here" link that jumps to the precise transcript moment (the command, the correction, the file-edit) the observe loop mined it from, plus a session↔proposals cross-reference both ways. Trace "this guardrail came from the session where I nearly deleted prod."
Basis
direct Observe runs at session-end (squash-merge → mine), and the Ribbon only shows a passive pending count after the user has left the terminal — no component listens for the live→dead session transition. The Overview's SESSIONS and THE BRAIN columns are visually severed with no path between them. external git bisect — making any point in history one jump from its cause; ported to "note → the moment it was born."
RationaleThe Work⇄Brain gap is a perception gap as much as a navigation one — the user doesn't feel observed while working, and never gets a "look what I taught it" moment. The most bridgeable instant is session-close: the user just did the work and is in a reflective state. Provenance is also the single strongest antidote to the "creepy black box" failure mode of silent observation — trust collapses if the user can't answer "why does it think this?"
DownsidesNeeds a daemon→client event on session-state transition (the turn/end hooks fire today but are discarded client-side). Linking notes to transcript offsets requires observe to record the source span (it has the data; it may not persist the pointer yet). Risk of a nagging close-card — make it dismissible and quiet.

5Setup as the brain's first proposals

A3 · setup → first valueConfidence 75%Complexity Medium
DescriptionReplace the linear setup Stepper with the project's first proposals, reviewed through the same gate the user will use forever: a just-opened folder shows a tiny review queue — "I'd like to: protect against secret reads · watch app.tsx · enable observe." Configuration enters through the one mechanism, so the user learns the moat on turn zero and never meets a second interaction grammar. The first real session then hands off to a narrated "here are the first 3 things I noticed about your code" — the docent walk that makes the cold-start aha land on the user's own work, not canned copy.
Basis
reasoned Setup and Review are two different interaction models (a checklist vs an approval queue) for the same thing — "human consents to a change to the project." zz init already ships only a guardrails floor and modules "materialize on demand," so setup is already a sequence of first proposals; the Stepper is a redundant wrapper. Every rung is deterministic — the only thing the human contributes is consent, which is exactly what the gate is for. external The museum docent's narrated first-three as the cold-start mental-model bootstrap.
RationaleOne interaction model instead of two; the user's first action and their thousandth are the same gesture (approve a proposal). It collapses the highest-drop-off surface and teaches the moat mechanic immediately — and the narrated first session is the only proof that converts (the loop demonstrated on the user's own code).
Downsides"Silent/auto setup" reads as spooky if over-done — keep each step a visible consent, never invisible. Enabling host hooks is a real system change the user should understand; the proposal copy must explain it. The narrated-first-session depends on observe producing legible first-run signal, which is itself cold.

6The brain as a "what I learned" feed

A2 · orientation & navConfidence 70%Complexity Medium-High
DescriptionOffer the Brain world a feed/timeline lens as the default, instead of tables-first. The brain presents as a reverse-chronological stream of "what I learned" cards — "Learned: app.tsx is the hot file · 3 sessions ago", "Promoted a guardrail: no secret reads" — narrative, scannable, weighted by corroboration so load-bearing knowledge renders bolder and one-off notes fainter (a "desire-path" map of where work actually flowed). The grid / table / ER views stay, one toggle away, as the power lens. The database is the source of truth; the feed is the humane surface over it.
Basis
reasoned The locked design optimizes for the admin/CRUD model (tables-first), but a user's relationship to their project's brain is closer to "what does it know / what did it just learn" than "let me query a table." A grid answers retrieval; a feed answers awareness, and most of the time the user wants awareness. The append-only mutation log is already chronological and timestamped — the feed is a reframe of existing data, not new data. external Campus "desire paths" — let usage reveal structure (corroboration as trail-weight) before committing to it.
RationaleReframes the brain from an admin tool you operate into a companion whose growth you witness — the emotional core of "watch a brain grow from your work," and plausibly a bigger retention lever than any single feature. The corroboration-weighting answers "which of these 200 notes actually matter?" at a glance, which a flat table never does.
DownsidesA second lens to build and keep in sync with the grid (more surface). A feed is worse than a grid for the actual editing/fixing task, so the toggle must be obvious and fast. Risk of feeling like a "toy" to power users if the grid is hard to reach.

7Identity from the brain — the project names itself

A5 · identity & personalizationConfidence 65%Complexity Medium
DescriptionReplace user-chosen project identity (an emoji + theme picked at creation) with an identity the project earns from its own brain: the emoji, color, and a one-line "what I am" descriptor are derived and proposed from what the brain has learned (a project that grew card-mechanics notes proposes 🃏 and "a card-game engine"). The user ratifies or overrides through the gate. Identity becomes another evolving, reviewed artifact rather than a cosmetic decision made at the moment the user knows least about the project.
Basis
reasoned Identity is currently a user input at creation — made when the user knows least — yet the brain accumulates exactly the signal that defines the project (hot files, dominant module, recurring vocabulary). Letting the project name and color itself from its own knowledge makes personalization a consequence of work, and routes even identity through observe→propose→review — reinforcing the gate as the one universal grammar (cousin to idea #5). The per-project emoji/health data and the deterministic-default mechanism already exist as the seam.
RationalePersonalization stops being a setup tax and becomes a delightful payoff — "my project named itself" — and demonstrates the loop's reach by applying it to the project's own identity. A small, memorable proof that the brain understands the project.
DownsidesThe smallest, most "gimmick-risk" bet of the set — some users want to name their own things and will find auto-naming presumptuous (override must be trivial). Depends on enough brain signal to derive a non-generic identity, so it's empty until real work accrues. Lower confidence; best as a delight layer once #1/#5 land.
